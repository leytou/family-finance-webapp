# 收入 / 支出 / 结余 口径重分类 设计

日期：2026-06-14

## 背景与目标

当前月度表「收入」只含手填进账项，「结余」公式为 `收入 − 支出 + 理财 + 公积金转入`，分类不直观。用户希望：把**理财收益**和**公积金提取**归进**收入**，让**结余 = 收入 − 支出**直接成立。

核心约束：**最终存款（累计储蓄）数字不得变化**——这只是收入/支出/结余三列的分类口径调整，不改变任何资金的实际走向。

## 公积金「月冲」的处理（已与用户确认）

公积金每月自动帮还房贷的「月冲」**不算收入**，而是**抵掉等额房贷**——支出里的房贷只记自己实际掏钱还的部分（即现有「存款补扣」= 房贷全额 − 月冲）。这样「收入 − 支出」正好等于银行卡当月真实进出。

## 月度新口径

记号：

- P = 手填进账项之和（正的现金流列，含专项正项）
- N = 手填开支项绝对值之和（负的现金流列，含专项负项）
- M = 房贷月供绝对值（公积金专区，无公积金时为 0）
- O = 公积金月冲（fundOffset）
- W = 公积金提取（fundWithdrawal）
- I = 理财收益（investReturn）
- shortfall = 存款补扣 = max(0, M − O)

| 字段 | 新定义 |
|---|---|
| 收入 monthlyIncome | P + I + W |
| 支出 monthlyExpense | N + shortfall |
| 结余 monthlyBalance | 收入 − 支出 |

### 等价性验证

新结余 = (P + I + W) − (N + M − O) = P + I + W + O − N − M

旧结余 = totalFlow + investReturn + fundOutflow = (P − N − M) + I + (O + W) = P + I + W + O − N − M  ✓ 相等

→ 累计储蓄（cumSavings = prevCum + monthlyBalance）数值完全不变。

## 公式弹窗文案（utils/formula.ts）

- **收入**：`收入 = <正项明细> + 理财(I) + 公积金提取(W)`（值为 0 的项不显示；全部为 0 则 `收入 = 0`）
- **支出**：`支出 = <负项明细> + 存款补扣(shortfall)`（房贷由「全额」改为「存款补扣」；shortfall = 0 时不显示该项）
- **结余**：`结余 = 收入(monthlyIncome) − 支出(monthlyExpense)`（去掉 + 理财 + 公积金转入）

## 年度表（AnnualTable.vue）

1. **新增「年收入」「年支出」两行**（紧邻「年度结余」，便于对照「收入 − 支出 = 结余」）。
   - 年收入 = 该年各月 monthlyIncome 之和
   - 年支出 = 该年各月 monthlyExpense 之和
   - `YearSummary` 类型新增 `yearIncome` / `yearExpense` 字段，computed 中计算。
2. **年度结余 hover 公式**更新为新口径表述（年度结余 = 年收入 − 年支出）。
3. 年度结余数值不变（= 各月 monthlyBalance 之和），自动正确。

## 连带自动对齐（无需改逻辑）

- **图表（financeChart.ts）**：收入/支出柱、净结余均读 monthlyIncome / monthlyExpense，自动跟随；「收入 − 支出 = 净结余」关系仍成立。
- **多方案对比（useComparison.ts）**：totalIncome / totalExpense / netSavings 自动跟随。
- **按年聚合（aggregateByYear）**：income / expense 基于 monthlyIncome / monthlyExpense，自动跟随。

## 「理财」列保留（已与用户确认）

- 月度表「理财」列、年度表「理财收益」行**保留**，单独展示理财收益；收入里也含它，不重复计算（结余只从收入/支出推）。
- 公积金提取仍无独立列，在公积金余额编辑器输入，金额体现在收入（hover 收入公式可见）。
