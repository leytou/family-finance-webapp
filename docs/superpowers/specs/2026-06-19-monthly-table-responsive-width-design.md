# 月度流水自适应窗口宽度 · 设计文档

- 日期：2026-06-19
- 状态：已与用户确认方案，待写实现计划
- 涉及组件：`MonthlyTable.vue`

## 一、背景

用户反馈：月度流水表不像年度汇总那样「随网页宽度变化自适应」——拉宽/收窄浏览器窗口时，月度流水表宽度固定不变、底部常驻横向滚动条；年度汇总表则能跟着窗口伸缩。

## 二、根因（实测确认）

月度流水与年度汇总使用**同一套**「铺满 + 滚动」机制：

- 表格容器 `overflow-auto`，`<table class="min-w-full">`，单元格 `whitespace-nowrap`。
- 表格实际宽度 = `max(容器宽度, 所有列 min-content 之和)`。
  - 列 min-content 之和 ≤ 容器宽度 → 表格铺满容器、随窗口伸缩（「自适应」）。
  - 列 min-content 之和 > 容器宽度 → 表格固定为列之和、横向滚动、窗口变化时不动（「不自适应」）。

两者差异**仅在列数与列内容宽度**：

- 年度汇总：列少（项目 + 每年一列），min-content 之和几乎总小于容器宽度 → 永远铺满自适应。
- 月度流水：列多（月份 + N 个动态收支列 + 添加列占位 + 专项 + 理财 + 收入 + 支出 + 结余 + 存款 + 公积金专区 5 列 + 对比 2 列），min-content 之和容易超过容器宽度 → 落入滚动区间、不随窗口变。

实测（买房方案：8 收支项 + 公积金 = 21 列，1280 视口）：

- 21 列 min-content 之和 ≈ **1242px**。
- 容器（去 `px-8` 边距 + 滚动条）≈ **1201px**。
- 1242 > 1201 → 装不下、滚动、不随窗口变。
- 年度汇总 7 列 min-content 之和 ≈ 855px < 1201 → 铺满自适应。

即：用户「月度流水不自适应」的根因是**列太多撑爆窗口**，而非机制缺失。

## 三、方案（已与用户确认）

**收紧月度流水列内边距 + 辅助紧凑，降低 min-content 之和，使常见笔记本窗口（1280–1366）下、典型列数（18–22）能装得下、铺满自适应；装不下才滚动。**

用户在备选方案中明确否决了「截断数字换永远铺满」（要求数字完整可读），接受「装不下才滚动」。

## 四、详细改动

仅改动 `src/components/MonthlyTable.vue`（年度汇总、其他视图不动）。

1. **收紧数据单元格水平内边距**：将表格内 `td`/`th` 的 `px-1`（左右各 4px）收紧（具体值以实测装得下为准；实测 4px→1px 在 21 列场景省约 41px，恰好让 1280 容器装下）。
2. **辅助紧凑（按实测需要启用）**：
   - 「添加新列」表头下的空占位 `td`（`px-1 py-0`）收窄或去除多余内边距。
   - 列头启用/禁用按钮、删除按钮的 `mr-1` / `ml-1` 间距酌情收紧。
3. **保持铺满机制**：`min-w-full` + `table-layout: auto` 不变，确保装得下铺满、装不下滚动。

验收临界点（实测）：18–22 列场景下，1280px 窗口（容器约 1201px）能装下、无横向滚动条；窗口变宽时表格铺满跟随伸缩。

## 五、不改动范围

- 年度汇总 `AnnualTable.vue`、对比视图、图表、计算器：不动。
- 所有计算逻辑（`useCalculation`）、状态（`useStore`）、公式、数据模型：不动。
- 单元格交互（编辑、右键菜单、拖拽列、hover 公式）：行为不变，仅内边距视觉收紧。

## 六、验证方式

此为纯样式/布局调整，浏览器布局行为无法用 jsdom 单测断言，**以浏览器多分辨率实测为准**：

1. 导入接近真实的「买房方案」数据（多收支项 + 公积金，约 21 列）。
2. 在 1280 / 1366 / 1536 / 1920 四个视口宽度下，用脚本测量月度流水 `table.scrollWidth` 与滚动容器 `clientWidth`，确认：
   - 1280–1366：`scrollWidth ≤ clientWidth`（装下、铺满、无滚动条）。
   - 表格宽度随窗口伸缩（窗口变宽 → 表格变宽铺满）。
3. 列继续增多（如 ≥25）或窗口更窄时，确认仍能正常滚动（不溢出、不破版）。
4. 改动前后截图对比，供用户确认视觉紧凑度可接受。

## 七、边界与取舍

- 收紧内边距会让数字两侧留白变窄、整体更紧凑；财务表格常见，不影响读数。
- 若列数极多（25+）或窗口很小，收紧后仍可能装不下、滚动——物理限制，本次不解决（用户已接受滚动）。
- 如未来确需在极窄窗口下也铺满，再评估「折叠次要列（公积金/对比）」方案（用户本次未选）。
