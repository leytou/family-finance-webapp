# 专项事件列 — 设计规格

## 背景

家庭财务规划中，买房、生育、换车、择校费等是**单月一次性大额收支**。当前要表达这类事件，只能「加一列」：为每个事件新建一个 `FlowColumn`，整列横跨 60 个月却只有 1~2 个格子有值。

核心痛点：**一列只填一格，太占横向空间**。空列把右侧汇总列（理财/收入/支出/结余/存款）挤出屏幕，违背「一屏看全」原则。

此外，普通列的值会**向后携带延续**（`resolveColumnValue` 规则 2），一次性支出会被误带到之后所有月份，表达真正的单次事件很别扭。

## 目标

- 用**固定一列**「专项」容纳所有单月一次性大额事件，列数从「每事件一列」降到「固定一列」。
- 同月多笔事件在同一格内呈现：显示净额 + 笔数角标，点开看明细并增删改。
- 事件天然是单月脉冲，不携带延续，顺带消除一次性支出被蔓延的隐患。
- 持续性收支（工资、房租、教育学费等）仍走普通列；事件只管单月一次性大额。两者职责清楚。

## 非目标（YAGNI）

- 事件**不支持时间区间**（起始月→结束月）。子女教育等持续性支出用普通列 / 年度同步表达。
- 事件**不可启用/禁用、不可拖拽、不可改名/删除整列**——「专项」是固定派生列。
- 不为专项列接入右键「同步到下方每年此月 / 清除下方编辑值」（脉冲语义不适用）。
- 不做全局事件列表/时间轴视图。

## 方案选型

选定 **方案 A**：独立 `events` 数据 + 派生「专项」虚拟列。

排除项：
- **复用 FlowColumn 存净额**：丢失每笔事件名称；且携带规则会把一次性值延续到之后每月，需硬塞 0 或误用年度标记，别扭。
- **事件支持区间**：表达力最强但过度设计，YAGNI。

## 数据模型（src/types.ts）

新增 `MilestoneEvent`，并在 `PlanData` 增加 `events` 字段：

```typescript
// 单月一次性大额收支（买房、生育、换车、择校费……）
export interface MilestoneEvent {
  id: string
  name: string      // 如"买房"
  month: number     // YYYYMM
  amount: number    // 元；正=收入（如奖金/卖房），负=支出（如买房）
}

export interface PlanData {
  version: number          // 保持 2
  systemParams: SystemParams
  columns: FlowColumn[]
  anchors: MonthlyAnchor[]
  snapshots: PlanSnapshot[]
  events: MilestoneEvent[] // 新增
}
```

- `version` 保持 2。旧数据（无 `events` 字段）在加载时由 `normalizeWorkspace` 补 `[]`，graceful 兼容，无需正式迁移逻辑。

## 计算引擎集成（src/composables/useCalculation.ts）

在 `calculate()` 逐月循环中，普通列经 `resolveColumnValue` 解析后，**仅当该月有事件**时注入一个虚拟列值：

```typescript
const EVENT_COLUMN_ID = '__events__'

// ...现有：const columnValues = plan.columns.map(col => resolveColumnValue(col, month))

const monthEvents = plan.events.filter(e => e.month === month)
if (monthEvents.length > 0) {
  const eventsNet = monthEvents.reduce((s, e) => s + e.amount, 0)
  columnValues.push({
    id: EVENT_COLUMN_ID,
    name: '专项',
    amount: eventsNet,
    isEdited: false,   // 派生值，非用户直接编辑的单元格
    enabled: true,
  })
}

const activeValues = columnValues.filter(col => col.enabled !== false)
// totalFlow / monthlyIncome / monthlyExpense / monthlyBalance / cumSavings 逻辑不变
```

**关键决策：专项计入收入/支出。** 因为虚拟值进入 `columnValues` → `activeValues`，所以：
- 正数事件计入 `monthlyIncome`，负数事件计入 `monthlyExpense`（绝对值）。
- `totalFlow`、`monthlyBalance`、`cumSavings` 自然包含专项。
- 买房 -200 万会让该月「支出」跳到约 202 万、存款同步降 200 万——语义诚实，实现最简。

事件**不经 `resolveColumnValue`**，因此携带延续逻辑完全不影响事件——脉冲天然不蔓延。

## 月度表「专项」列（src/components/MonthlyTable.vue）

### 列位置与表头

```
月份 | [用户列1] [用户列2] ... [+] | 专项 | 理财 | 收入 | 支出 | 结余 | 存款
```

- 「专项」紧跟用户列和「+」之后、汇总列（理财…）之前。
- 用 `border-l` 与汇总区隔开（同现有「理财」的分隔线一致），作为「输入区」最后一列。
- 表头为固定文字「专项」，右对齐 `tabular-nums`，**不可拖拽 / 改名 / 删除**。

### 单元格渲染

数据来源：直接读 `store.data.value.events`（不依赖 `columnValues`，便于拿到笔数与明细）。新增一个按月分组的 computed：

```typescript
const eventsByMonth = computed(() => {
  const map = new Map<number, MilestoneEvent[]>()
  for (const e of store.data.value.events) {
    const arr = map.get(e.month) ?? []
    arr.push(e); map.set(e.month, arr)
  }
  return map
})
```

逐月单元格逻辑：
- `count === 0`：空白（不显示 0），仍可点击打开编辑器添加第一笔。
- `count >= 1`：显示 `formatCurrency(净额)`，给淡色背景（区别于普通编辑格的蓝底 `bg-blue-100`），负数斜体。
- `count >= 2`：在金额后追加角标 `·N`（如 `-220万 ·2`，蓝色，类似年度标记 `↻` 的小角标样式）。

**注意**：v-for 渲染用户列时调用 `getColumnValue(result, column.id)`，虚拟 `__events__` 值不在 `store.columns` 中，不会被该循环渲染——避免重复。专项格由专门的 `<td>` 渲染。

### 交互

- 点击任意专项格（空或有值）→ 打开事件编辑器，锚定到该格。
- 专项格不接入 `@contextmenu`（无年度同步 / 清除下方语义）。

## 事件编辑器（新组件 src/components/EventEditor.vue）

仿 `FormulaPopover` / `ContextMenu` 的绝对定位弹层。

### Props / 交互

- 接收 `month`、定位坐标 `{ x, y }`、该月初始事件列表。
- 顶部：月份标题（如「2028年6月 专项」）。
- 列表：每行 = `[名称输入] [金额输入] [× 删除]`。
- 底部：`+ 添加` 追加空行。
- **提交模型：本地草稿编辑，关闭时整体写回。** 组件持有 reactive 草稿数组；关闭（点击外部 / Esc / 「完成」按钮）时调用 `store.replaceMonthEvents(month, draft)`。
- 半空行（名称为空或金额非有限数）在提交时静默丢弃。

### useStore 新增

```typescript
// 整体替换某月的事件：先删该月全部，再写入有效项
function replaceMonthEvents(month: number, items: { name: string; amount: number }[]): void {
  const plan = getActivePlan()
  plan.events = plan.events.filter(e => e.month !== month)
  for (const it of items) {
    const name = it.name.trim()
    if (name && Number.isFinite(it.amount)) {
      plan.events.push({ id: generateId(), name, month, amount: Math.round(it.amount) })
    }
  }
}
```

v1 仅需 `replaceMonthEvents`（编辑器唯一入口）；不为单条增删另开 API（YAGNI）。

## 年度表（src/components/AnnualTable.vue）

**零改动。** `sumColumnValues` 遍历 `result.columnValues` 按列名求和，专项虚拟值已在内 → 自动出现一行「专项」年度合计。`getColumnTotal` 对无事件年份返回 0。

**已知限制**：年度表按列名聚合，若用户把某个普通列也命名为「专项」，该列金额会与事件净额合并到同一行。概率极低，v1 接受。

## 持久化与校验（src/composables/useStore.ts）

- `createDefault()`：plan 增加 `events: []`。
- `normalizeWorkspace()`：缺 `events`（或非数组）时补 `[]`，与 `initialDeposit` 同样的轻量容错。
- `isValidPlanData()`：`events` 存在时校验为数组且每项合法（`id`/`name` 为字符串、`month` 为整数、`amount` 为有限数），仿 `isValidColumn`；缺失则放行。
- 导出/导入：**无需改 useFileIO**——导出整对象序列化 `workspace.value`（events 自动带上），导入只校验外壳，深层校验落在 `isValidPlanData`。

## 边界情况

- 事件金额为 0：合法，照常显示（金额 0、计入 0）。建议用户避免，但不阻止。
- 同月同名多笔（如两次「换车」不同月；同月同名两笔）：允许，各自独立条目。
- 月份超出规划区间（< startMonth 或 ≥ 60 个月后）：`calculate` 只遍历规划期 60 个月，区间外事件不参与计算但仍存储；编辑器只从规划期内月份打开，不会产生区间外事件。可接受。
- 删除某月全部事件：编辑器清空所有行后关闭 → `replaceMonthEvents` 删尽 → 单元格回归空白。
- 多方案：`events` 随 `plan` 走，每方案独立（`getActivePlan()` 已隔离）。

## 测试清单（TDD）

### useCalculation（calculation.spec.ts）
- 单月单笔事件：净额正确计入 `totalFlow` / `monthlyIncome`（正）或 `monthlyExpense`（负）/ `cumSavings`。
- 单月多笔事件：净额为各笔之和。
- 无事件月：不注入虚拟值，贡献 0。
- 正数事件计入收入、负数事件计入支出。
- 跨年事件：年度聚合正确。
- 事件不影响普通列携带延续逻辑（普通列值仍按规则延续）。

### useStore（store.spec.ts）
- `replaceMonthEvents`：新增 / 修改 / 删除某月事件后写回正确。
- 半空行（无名称 / 非有限金额）被丢弃。
- 旧数据无 `events` 字段：加载后默认 `[]`。
- 非法 `events`（缺字段 / 类型错）被 `isValidPlanData` 拒绝。

### MonthlyTable（组件测试，可选）
- 无事件月：专项格空白。
- 有事件月：显示净额；≥2 笔显示 `·N` 角标。
- 点击专项格打开编辑器。

### EventEditor（组件测试，可选）
- 预填该月已有事件。
- 增 / 删 / 改后提交，调用 `replaceMonthEvents` 写回。
- 关闭不提交半空行。
