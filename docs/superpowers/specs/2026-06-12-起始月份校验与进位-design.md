# 起始月份校验与进位规范化设计

> 日期：2026-06-12
> 状态：已确认，待实现

## 背景

「起始月份」输入框（`src/App.vue`）当前用 `v-model.number="data.systemParams.startMonth"` 直接双向绑定，**没有任何校验**。用户可输入 `202613`（13 月）、`202600`（0 月）、`2026`（位数不足）、负数、字母等非法值。

`src/utils/month.ts` 的 `addMonths` 内部已能正确跨年进位（如 `202613` 会被当成 `202701` 参与计算），但**只在计算时隐式生效**：

- 存储的 `startMonth` 仍是非法值（如 `202613`）。
- `formatMonth(202613)` 显示成 `"2026-13"`，丑陋且错误。
- `addSnapshot` 用 `formatMonth(plan.systemParams.startMonth)` 生成快照名，会带上非法月份。
- `startMonth` 作为后续 60 个月投影的基准，非法基准会让整个月份序列错位。

此外，`addMonths` 的进位规则是按 `year=floor(V/100)、month=V%100` 拆分后做 `年*12+月` 的数学进位。这意味着位数不足的值会产生**非预期的年份**：如 `2026` 被拆成 `year=20, month=26`，进位后变成 `2202`，并非用户想要的 2026 年。

`month.ts` 目前没有 `normalize`/`validate` 之类的通用月份工具函数。

## 目标

1. 起始月份输入框的值在离开输入框（blur 或回车）后，**始终是合法的 YYYYMM**（6 位整数、月份 1–12）。
2. 月份越界（13 月、0 月）时按进位规则自动转成合法月份并写回（如 `202613→202701`、`202600→202512`）。
3. 完全非法的输入（位数不足、负数、非数字、清空）不写入数据，回退到上一次合法值。
4. 在 `month.ts` 提供可复用、可单测的纯函数，规范化逻辑集中、单一受控入口。

## 需求澄清结论

- **处理策略**：输入即规范化进位（非「拒绝提示」，也非「拆分年/月控件」）。
- **边界规则**：严格要求 6 位整数 YYYYMM（年份隐含 1000–9999），否则视为非法、回退旧值。位数不足（`2026`、`20261`）一律不进位，避免被错误解释成怪异年份。
- **触发时机**：blur 或 Enter。实时校验会让用户打到第 4、5 位时就被回退、无法输完 6 位，已排除。
- **失败反馈**：输入框短暂红框（1.5s）；显示值靠单向绑定自动回退到旧合法值，无需手动恢复。
- **作用范围**：仅起始月份输入框。表格编辑、锚点等其它月份入口不产生自由文本非法值，不在本次范围内。

## 方案对比

- **A：纯工具函数 + 组件内联** —— `month.ts` 加 `normalizeMonth`；`App.vue` 在 input 的 blur/Enter 里处理写回与回退。改动最小，但规范化逻辑耦合在组件，未来第二个改 `startMonth` 的入口要重复写。
- **B（选定）：工具函数 + useStore 受控动作** —— `month.ts` 加 `normalizeMonth` 纯函数；`useStore` 暴露 `setStartMonth(raw)` 作为受控入口；`App.vue` 在 blur/Enter 时调用。逻辑集中、单一受控入口、纯函数易测，符合现有 store 模式。
- **C：computed getter/setter 拦截** —— 隐式、调试难、与「失焦才规范化」的时机不好协调，排除。

## 选定设计（方案 B）

### 1. 工具函数层（src/utils/month.ts）

新增两个纯函数，复用已有 `addMonths` 的进位规则（DRY）：

```ts
/** 是否为 6 位整数 YYYYMM（年份隐含 1000–9999）；只校验位数，月份越界交给 normalizeMonth 进位 */
export function isValidYyyyMm(value: number): boolean {
  return Number.isInteger(value) && value >= 100000 && value <= 999999
}

/** 规范化 YYYYMM：6 位整数则按「年*12+月」原地进位（202613→202701、202600→202512）；否则返回 null */
export function normalizeMonth(value: number): number | null {
  return isValidYyyyMm(value) ? addMonths(value, 0) : null
}
```

`addMonths(value, 0)` 即「原地进位」：合法值（月份 1–12）原样返回，越界月份（13、0、99 等）进位为合法月份，完全复用现有进位逻辑。

行为示例：

| 输入 | `normalizeMonth` 结果 | 说明 |
|---|---|---|
| `202612` | `202612` | 合法，原样 |
| `202613` | `202701` | 13 月进位为次年 1 月 |
| `202600` | `202512` | 0 月进位为上年 12 月 |
| `2026` | `null` | 位数不足 |
| `20261` | `null` | 位数不足 |
| `-5` / `0` | `null` | 非 6 位正整数 |
| `1.5` / `NaN` | `null` | 非整数 / 非数字 |

### 2. 状态管理层（src/composables/useStore.ts）

新增受控动作，作为修改 `startMonth` 的唯一入口：

```ts
/** 设置起始月份：合法（含进位）写入并返回 true；非法（非 6 位整数）忽略、返回 false */
function setStartMonth(raw: number): boolean {
  const normalized = normalizeMonth(raw)
  if (normalized === null) return false
  getActivePlan().systemParams.startMonth = normalized
  return true
}
```

在 `useStore` 的返回对象中导出 `setStartMonth`。进位后的值（如 `202613→202701`）写入的是规范化结果；非法值完全不改写现有 `startMonth`，并返回 `false` 供调用方做 UI 反馈。

作用对象为当前激活方案（与其它列/锚点操作一致，复用 `getActivePlan()`）。

### 3. 组件层（src/App.vue）

起始月份 `input` 从 `v-model.number`（无校验）改为受控 + 失焦规范化：

```vue
<input
  :value="data.systemParams.startMonth"
  @blur="onStartMonthBlur"
  @keydown.enter="($event.target as HTMLInputElement).blur()"
  type="number"
  :class="startMonthInvalid ? 'border-red-500' : 'border'"
  class="rounded px-2 py-1 text-sm w-24"
  placeholder="YYYYMM"
/>
```

```ts
const { data, setStartMonth } = useStore()
const startMonthInvalid = ref(false)

function onStartMonthBlur(e: Event) {
  const raw = Number((e.target as HTMLInputElement).value)
  if (!setStartMonth(raw)) {
    startMonthInvalid.value = true
    setTimeout(() => (startMonthInvalid.value = false), 1500) // 短暂红框反馈
  }
  // data.startMonth 未变时，:value 单向绑定自动把输入框回退到旧合法值
}
```

要点：
- `:value` 单向绑定。blur 后若规范化成功，`setStartMonth` 已写入新值，`:value` 反映新值；若失败，`data.startMonth` 未变，`:value` 自动回退到旧合法值，无需手动恢复 DOM。
- 清空输入框时 `Number("")` = `0`，`normalizeMonth(0)` = `null` → 回退旧值。
- `@keydown.enter` 触发 blur，进而触发 `@blur`，保证回车也规范化。

### 4. 测试（TDD，先红后绿）

**新增 `tests/utils/month.spec.ts`**（新文件），覆盖 `normalizeMonth` / `isValidYyyyMm`：
- 合法原值原样返回（`202612`）。
- 月份越界进位（`202613→202701`、`202600→202512`、`202699`）。
- 位数不足返回 `null`（`2026`、`20261`、`202`）。
- 负数 / `0` / 非整数 / `NaN` 返回 `null`。
- 边界值 `100000`、`999999` 的行为（合法 6 位整数，进位后可能溢出 6 位，不专门约束，仅记录行为）。

**追加 `tests/composables/useStore.spec.ts`**，覆盖 `setStartMonth`：
- 合法原值写入、`getActivePlan().systemParams.startMonth` 更新、返回 `true`。
- 越界月份进位后写入规范化值（`202613` → 存为 `202701`）。
- 非法值返回 `false`、原 `startMonth` 不变。
- 作用于当前激活方案（切换方案后只影响激活方案）。

`App.vue` 的 blur 规范化属组件交互层，单测成本较高，以工具函数 + store 动作覆盖为主；如现有组件测试基建便利，可补一条「非法输入后回退」的轻量断言。

### 5. 非目标（YAGNI）

- **不迁移**已有 `entries` / `anchors` / `snapshots` 的月份 key——它们以绝对 YYYYMM 存储，改 `startMonth` 不改变其语义，仅改变投影起点。
- 不对年份额外设上下限（6 位整数校验已隐含 1000–9999；极端进位溢出如 `999999→1000702` 不专门处理）。
- 不改表格编辑、锚点等其它月份输入入口。
- 不做实时规范化、不做 toast/弹窗级错误提示（红框已足够轻量）。
